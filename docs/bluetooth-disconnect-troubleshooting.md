# Диагностика самопроизвольных отключений Bluetooth-устройств (Linux)

Практическое руководство: **почему BT-устройство (мышь, клавиатура, гейм-пад,
колонка, наушники) само отваливается, когда им не пользуются, и как найти
настоящую причину, а не первую попавшуюся.**

Документ написан по итогам реального разбора двух машин (см. раздел
[«Журнал реальной диагностики»](#журнал-реальной-диагностики)). Главная ценность —
не готовый ответ, а **методика**: как проверять теории и как не обмануть самого
себя.

> Читать можно с нуля: команды самодостаточны, их можно копировать целиком.
> Почти все диагностические команды **безопасны** (только читают состояние).
> Команды, которые что-то меняют, помечены **⚠️ ИЗМЕНЕНИЕ**.

---

## TL;DR — порядок действий

1. **Сначала думай, потом трогай.** Любой рестарт сервиса или `connect` сам
   вызывает разрыв/переподключение. Если начать «чинить» до того, как понял
   проблему, — будешь мерить шум, который создал сам. (Мы на это наступили.)
2. **Собери вводные** — что за устройство и что за контроллер → [Шаг 0](#шаг-0--собрать-вводные).
3. **Поймай код причины разрыва** через `btmon` → [Шаг 1](#шаг-1--поймать-код-причины-разрыва-главное). Это единственный
   факт, который сразу отсекает половину теорий.
4. По коду причины иди в [дерево решений](#дерево-решений--по-коду-причины).
5. Каждую теорию **проверяй экспериментом** (delta-тест, A/B), а не «на глаз».
6. Фикс считается доказанным, только если после него разрывы прекратились
   при **пассивном** наблюдении (ты ничего не дёргаешь).

---

## Немного теории (чтобы понимать, что ищем)

У Bluetooth-соединения есть несколько независимых уровней, и «отвал» может
происходить на любом из них:

| Уровень | Кто отвечает | Как ломается |
|---|---|---|
| **Радиолинк (ACL)** | BT-контроллер + прошивка устройства | Таймаут связи (RF-помехи, дальность, сон устройства) |
| **USB-контроллер** | ядро Linux (драйвер `btusb`) | Контроллер уходит в USB-автосуспенд и перестаёт обслуживать линк |
| **Питание устройства** | само устройство | Дешёвые колонки/HID сами выключаются в простое ради экономии батареи |
| **Аудио-стек** (только звук) | PipeWire / PulseAudio / WirePlumber | Усыпляет неактивный аудио-узел, закрывает A2DP-транспорт |

**Ключевая мысль:** «устройство отвалилось» — это симптом. Причина может быть на
любом уровне, и лечится каждый по-своему. Поэтому нельзя сразу ставить «фикс из
интернета» — сначала надо понять, **какой уровень** ломается.

Два больших класса причин:

- **Host-side** (виноват компьютер) — чинится и переносится на другие машины.
- **Device-side** (виновато само устройство) — с хоста почти не лечится;
  например, дешёвая колонка сама вырубается в тишине.

Цель диагностики — честно определить, какой это класс.

## Шаг 0 — Собрать вводные

Узнаём: что за устройство, что за BT-контроллер, как подключено.

```bash
# Что за устройства спарены/подключены (запомни MAC нужного)
bluetoothctl devices

# Подробности по конкретному устройству (подставь свой MAC)
bluetoothctl info AA:BB:CC:DD:EE:FF
```

Что смотреть в `info`:
- `Connected: yes/no`
- `Icon:` / `UUID:` — по ним понятно, **что это за класс**:
  - `Human Interface Device` → мышь/клава/пульт/геймпад (**HID**)
  - `Audio Sink` / `Advanced Audio` → колонка/наушники (**A2DP-аудио**)
- `Trusted: yes` — значит хост будет **автоматически переподключать** его.
  Это важно: при `Trusted: yes` быстрый цикл «отвал → сразу реконнект» —
  нормальное поведение bluez, а не поломка.

Теперь контроллер:

```bash
# Какой BT-контроллер и по какой шине
hciconfig -a 2>/dev/null | grep -E "hci|BD Address|UP RUNNING|Bus"
lsusb 2>/dev/null | grep -i blue
```

Производитель контроллера важен: **Intel** и **Realtek/MediaTek** ведут себя
по-разному с автосуспендом (Realtek исторически проблемнее).

## Шаг 1 — Поймать код причины разрыва (ГЛАВНОЕ)

`btmon` показывает событие `Disconnect Complete` и **код причины**. Этот код
сразу говорит, кто разорвал связь — устройство, радиолинк или хост.

### Как запускать

```bash
# Живой разбор (нужен root). Оставь устройство В ПРОСТОЕ и жди разрыва.
sudo btmon | grep --line-buffered -A3 -iE "disconnect complete|device disconnected"
```

Когда произойдёт разрыв, появится блок:

```
> HCI Event: Disconnect Complete (0x05) plen 4
        Status: Success (0x00)
        Handle: 2048 Address: AA:BB:CC:DD:EE:FF
        Reason: Connection Timeout (0x08)
```

Смотри на строку **`Reason:`**.

### ⚠️ Gotcha: btmon падает с `*** buffer overflow detected ***`

На части версий bluez текстовый парсер `btmon` крашится на определённых пакетах
(например длинных RFCOMM/`AT+...` строках от гарнитур). Живой разбор тогда
невозможен. Обход — **писать сырой дамп** (без парсинга на лету, не падает):

```bash
# 1) Пишем сырой захват в файл. Оставь работать, лови разрыв, потом Ctrl-C.
sudo btmon -w /tmp/bt.snoop
```

```bash
# 2) Декодируем файл ПОСЛЕ (варианты — что есть в системе):

# вариант A: обычным btmon из файла (может снова крашнуться, но нужные строки
#            обычно успевают вывестись до «плохого» пакета)
sudo btmon -r /tmp/bt.snoop 2>/dev/null | grep -A3 -iE "disconnect complete"

# вариант B: через tshark (надёжнее, если установлен wireshark/tshark)
sudo tshark -r /tmp/bt.snoop -Y "bthci_evt.code==0x05" \
     -T fields -e bthci_evt.bd_addr -e bthci_evt.reason
```

Если ничего из этого не срабатывает — заархивируй `/tmp/bt.snoop` и отдай тому,
кто разбирается (файл в стандартном формате BTSnoop, открывается Wireshark).

### Фоновый захват «поставил и забыл» (когда разрыв редкий)

Если устройство отваливается **редко** и караулить его с открытым терминалом
непрактично — запусти постоянный фоновый захват. Он переживёт выход из ssh
и сам запишет причину, когда разрыв наконец случится.

```bash
# Запустить фоновый захват (transient-сервис systemd, живёт после logout)
sudo systemd-run --unit=btmon-cap --collect btmon -w /var/log/bt-cap.snoop
```

Дальше живёшь как обычно, ничего не сторожишь. Когда заметишь разрыв (или просто
через день) — раскодируй накопленное:

```bash
sudo btmon -r /var/log/bt-cap.snoop 2>/dev/null | grep -A3 -iE "disconnect complete"
```

Остановить и убрать:

```bash
sudo systemctl stop btmon-cap        # остановить захват
sudo rm -f /var/log/bt-cap.snoop     # удалить дамп
```

> При простое BT-трафика почти нет, поэтому файл растёт медленно и спокойно
> проживёт несколько дней. Захват — **временный** (transient-сервис): после
> перезагрузки не восстановится, надо запустить заново.

### Таблица кодов причины

| Код | Название | Что значит | Куда копать |
|---|---|---|---|
| `0x08` | Connection Timeout | Линк умер сам, никто не рвал | Радио/питание/сон устройства (чаще **device-side**), либо контроллер уснул |
| `0x13` | Remote User Terminated | Устройство **осознанно** отключилось | Авто-power-save устройства → **device-side** |
| `0x14` | Remote terminated (low resources) | У устройства кончились ресурсы | device-side |
| `0x15` | Remote terminated (power off) | Устройство выключилось | device-side |
| `0x16` | Local Host Terminated | Отключил **хост** (bluez/софт/ядро) | **host-side** — копать на компьютере |
| `0x22` | LMP Response Timeout | Устройство перестало отвечать | Радио/сон устройства |
| `0x3e` | Conn Failed to Establish | Не удалось поднять связь | Радио/дальность/помехи |

**Логика:** `0x13/0x14/0x15` и часто `0x08/0x22` → виновато **устройство** (лечить
с хоста почти нечем). `0x16` → виноват **хост** (есть что чинить).

## Шаг 2 — Проверить теорию «USB-автосуспенд контроллера»

Популярная причина: BT-контроллер на USB уходит в автосуспенд и перестаёт
обслуживать соединение. Но **это надо доказать, а не предположить.**

### Быстрый снимок

```bash
# Универсально находим путь USB-устройства контроллера (без хардкода)
BT=$(readlink -f /sys/class/bluetooth/hci0/device/..)
echo "path=$BT"
echo "control=$(cat $BT/power/control)  delay=$(cat $BT/power/autosuspend_delay_ms)ms  status=$(cat $BT/power/runtime_status)"
```

- `control=auto` → автосуспенд **разрешён** (но это ещё не значит, что он рвёт связь).
- `control=on` → автосуспенд запрещён.

### delta-тест (доказательство: спит ли контроллер ВО ВРЕМЯ соединения)

Это ключевой эксперимент. Устройство должно быть **подключено и в простое**.

```bash
BT=$(readlink -f /sys/class/bluetooth/hci0/device/..)
x=$(cat $BT/power/runtime_suspended_time); sleep 60; y=$(cat $BT/power/runtime_suspended_time)
echo "runtime_status=$(cat $BT/power/runtime_status)  delta=$((y-x)) ms"
```

Как читать:
- `delta > 0` → за минуту простоя контроллер **успел поспать** → автосуспенд
  реально активен при живом соединении → **вероятная причина**, есть смысл чинить.
- `delta = 0`, `status=active` → контроллер **НЕ спит** пока есть соединение →
  автосуспенд **не может** рвать этот линк → **теория опровергнута**, не трогай USB.

> На практике многие контроллеры (напр. Intel) **не** усыпляются, пока есть
> активное соединение — тогда огромный накопленный `runtime_suspended_time`
> относится к периодам, когда ничего не было подключено, и к разрывам отношения
> не имеет. Именно поэтому нужен delta-тест, а не абсолютное число.

### Ещё более строгий A/B-тест

```bash
BT=$(readlink -f /sys/class/bluetooth/hci0/device/..)

# Фаза A: запрещаем контроллеру спать, ждём простой 10-15 мин.
echo on | sudo tee $BT/power/control      # ⚠️ ИЗМЕНЕНИЕ (временное, до перезагрузки)

# ... если за это время НЕ отвалилось — держим дальше:

# Фаза B: возвращаем как было, снова ждём простой.
echo auto | sudo tee $BT/power/control    # ⚠️ ИЗМЕНЕНИЕ (возврат)
```

Если A стабильно держит, а B стабильно роняет — причина доказана.

### Фикс (если теория подтвердилась) — отключить автосуспенд для всех BT-контроллеров

Виновник обычно не общий USB-автосуспенд, а сам драйвер `btusb`
(параметр `enable_autosuspend=Y` по умолчанию). Проверить:

```bash
cat /sys/module/btusb/parameters/enable_autosuspend    # Y = драйвер сам включает автосуспенд
```

```bash
# ⚠️ ИЗМЕНЕНИЕ: запретить btusb включать автосуспенд (переносится на любую машину)
echo 'options btusb enable_autosuspend=0' | sudo tee /etc/modprobe.d/btusb-no-autosuspend.conf
sudo modprobe -r btusb && sudo modprobe btusb    # или перезагрузка
```

Проверка, что фикс применён:

```bash
cat /sys/module/btusb/parameters/enable_autosuspend                        # ждём N
BT=$(readlink -f /sys/class/bluetooth/hci0/device/..); cat $BT/power/control # ждём on
```

Откат:

```bash
sudo rm /etc/modprobe.d/btusb-no-autosuspend.conf
sudo modprobe -r btusb && sudo modprobe btusb
```

## Шаг 3 — Проверить теорию «аудио-стек усыпляет узел» (ТОЛЬКО для колонок/наушников)

Для A2DP-устройств есть отдельная host-side причина: PipeWire/WirePlumber по
таймауту усыпляет неактивный аудио-узел и закрывает A2DP-транспорт. Некоторые
колонки, потеряв поток, сами рвут связь/выключаются.

### Диагностика

```bash
# Какой аудио-сервер работает
pgrep -a pipewire; pgrep -a wireplumber; pgrep -a pulseaudio

# Текущее состояние и таймаут усыпления bluez-УЗЛА (устройство должно быть подключено)
pw-dump 2>/dev/null | python3 -c '
import sys,json
for o in json.load(sys.stdin):
    if o.get("type")=="PipeWire:Interface:Node":
        p=o.get("info",{}).get("props",{})
        n=str(p.get("node.name",""))
        if "bluez" in n:
            print(n,"| state:",o["info"]["state"],"| suspend-timeout:",p.get("session.suspend-timeout-seconds","<не задан>"))
'
```

- `state: suspended` в простое + `suspend-timeout: <не задан>` → используется
  дефолт (обычно **5 секунд**) → узел усыпляется.

### ⚠️ Важная проверка причинности

Усыпление узла **не обязательно** рвёт BT-линк! Проверь напрямую: усыпи узел
(просто не играй звук) и смотри, отвалилось ли **устройство**:

```bash
# Наблюдаем 3 минуты: рвётся ли СОЕДИНЕНИЕ, пока узел спит
dev=AA:BB:CC:DD:EE:FF
for i in $(seq 1 12); do
  c=$(bluetoothctl info $dev 2>/dev/null | grep -oE "Connected: (yes|no)")
  echo "$(date +%H:%M:%S)  $c"
  sleep 15
done
```

Если `Connected: yes` держится, пока узел `suspended` — значит **усыпление аудио
не рвёт связь**, и эта теория тоже мимо (у нас так и вышло на HOS301).

### Фикс (если подтвердилось) — не усыплять BT-узлы

Синтаксис зависит от версии WirePlumber:

```bash
wireplumber --version    # смотрим 0.4.x (lua) или 0.5.x (conf/spa-json)
```

**Для WirePlumber 0.4.x (lua).** Важно матчить именно **узел** (`node.name`
`bluez_output.*`), а не карту (`device.name` `bluez_card.*`) — иначе настройка
до узла не доедет (мы на это наступили):

```bash
# ⚠️ ИЗМЕНЕНИЕ
mkdir -p ~/.config/wireplumber/bluetooth.lua.d
cat > ~/.config/wireplumber/bluetooth.lua.d/51-no-suspend.lua <<'LUA'
table.insert(bluez_monitor.rules, {
  matches = {
    { { "node.name", "matches", "bluez_output.*" } },
    { { "node.name", "matches", "bluez_input.*" } },
  },
  apply_properties = { ["session.suspend-timeout-seconds"] = 0 },
})
LUA
systemctl --user restart wireplumber    # ⚠️ рестарт САМ вызовет разрыв/реконнект — см. предупреждение ниже
```

Проверка:

```bash
pw-dump 2>/dev/null | grep -i "suspend-timeout"    # ждём "session.suspend-timeout-seconds": 0 на узле
```

Откат:

```bash
rm ~/.config/wireplumber/bluetooth.lua.d/51-no-suspend.lua
systemctl --user restart wireplumber
```

## Дерево решений — по коду причины

```
Поймали Reason из btmon (Шаг 1)?
│
├─ 0x16 Local Host Terminated ──────────► ХОСТ рвёт связь. Копать на компьютере:
│                                          • Шаг 2 (USB-автосуспенд контроллера)
│                                          • Шаг 3 (аудио-стек, если это колонка)
│                                          • bluetoothd/системные политики питания
│
├─ 0x13/0x14/0x15 Remote terminated ────► УСТРОЙСТВО само отключилось (power-save).
│                                          Host-side лечится плохо. Варианты:
│                                          • отключить power-save в самом устройстве (если есть)
│                                          • для аудио — держать поток живым (Шаг 3)
│
├─ 0x08 Timeout / 0x22 LMP timeout ─────► Линк умер сам. Сначала исключи host-side:
│                                          • delta-тест контроллера (Шаг 2)
│                                          • delta=0 → значит device-side/радио:
│                                            питание устройства, дальность, помехи (2.4ГГц/USB3)
│
└─ 0x3e Failed to establish ────────────► Радио: дальность, помехи, слабый сигнал.
```

Если код причины получить **не удалось** (btmon недоступен) — идём эмпирически:
прогоняем Шаг 2 и Шаг 3, каждый с проверкой причинности. Если оба опровергнуты —
проблема почти наверняка **device-side** (само устройство).

## Частые ошибки методики (на них мы наступили)

1. **Чинить до того, как понял.** Рестарт `wireplumber` и `bluetoothctl connect`
   **сами** вызывают отвал/реконнект устройства. Если делать это во время
   «наблюдения», легко принять свой же артефакт за органическую проблему.
   → Правило: во время ловли разрыва **не трогай** ни сервисы, ни соединение.
2. **Привязка к чужому логу.** В журнале были ошибки `Failure in Bluetooth audio
   transport`, но **для другого устройства**. Всегда сверяй MAC/имя в строке лога
   с тем устройством, что диагностируешь.
3. **Абсолютные счётчики вместо дельты.** Большой `runtime_suspended_time` сам по
   себе ничего не доказывает — он копится, когда ничего не подключено. Нужен
   delta-тест **при живом соединении**.
4. **Настройка «не туда».** В WirePlumber правило на карту (`bluez_card.*`) не
   доезжает до узла — усыпление продолжается. Проверяй, что параметр реально сел
   (`pw-dump | grep suspend-timeout`), а не просто «файл создан».
5. **Считать теорию доказанной без проверки причинности.** «Узел усыпился»
   и «связь оборвалась» — разные события. Проверяй, что одно **вызывает** другое,
   наблюдая напрямую.

## Журнал реальной диагностики

Здесь — что конкретно проверяли на двух машинах и какие выводы из этого следуют.

### Машина A — `thinkpad` (Intel AX211), устройство **Ulanzi** (BLE HID, геймпад-пульт)

| Проверка | Результат | Вывод |
|---|---|---|
| `bluetoothctl info` | `Icon: input-gaming`, `HID`, `Trusted: yes` | BLE HID-устройство; хост его авто-переподключает |
| Контроллер | Intel AX211 `8087:0033`, USB, `control=auto`, `delay=2000ms` | Автосуспенд разрешён |
| `btusb enable_autosuspend` | `Y` | Драйвер сам включает автосуспенд, даже при TLP `USB_AUTOSUSPEND=0` |
| `btmon` при разрыве | `Reason: Connection Timeout (0x08)` | Линк умер сам |
| **delta-тест при соединении** | `delta=0, active` | **Контроллер НЕ спит при живом соединении → автосуспенд НЕ причина** |

**Вывод по A:** разрыв Ulanzi — **device-side**: BLE HID-устройство само засыпает
в простое (нормальное энергосбережение). Host-side фикс `btusb` здесь бы не помог,
его **не ставили**. Изменений на машине A **не вносилось**.

### Машина B — `projector` (Realtek `0bda:b850`), устройство **HOS301** (BT-колонка, A2DP)

| Проверка | Результат | Вывод |
|---|---|---|
| `bluetoothctl devices` | `HOS301` (A2DP, class `0x240418`) | Колонка |
| Аудио-сервер | PipeWire + WirePlumber **0.4.17** | Актуально для Шага 3 |
| Лог WirePlumber | `Failure in Bluetooth audio transport` — но для `1C:6E:4C` (**другое** устройство), не HOS301 | Ложный след, отброшен |
| WirePlumber suspend-timeout | по умолчанию (5с), узел в простое `suspended` | Узел усыпляется |
| Правило на карту (`bluez_card.*`) | на узле `suspend-timeout: <не задан>` | **Не доехало до узла — фикс был неэффективен** |
| Правило на узел (`bluez_output.*`) | на узле `suspend-timeout: 0` | Настройка села корректно |
| Наблюдение 3 мин с фиксом | `Connected=yes` всё время, узел `suspended`, но разрыв всё равно случился | **Усыпление узла НЕ рвёт связь → аудио-теория опровергнута** |
| **delta-тест контроллера при соединении** | `delta=0, active` | **Realtek тоже НЕ спит при соединении → USB-автосуспенд НЕ причина** |

**Вывод по B:** на данный момент **все три host-side теории опровергнуты**
(USB-автосуспенд, аудио-усыпление). Разрывы происходят на уровне BT-линка,
быстро (<15с) и с авто-реконнектом.

**Важное наблюдение:** как только диагностика **перестала трогать систему**
(никаких рестартов/`connect`), разрывы прекратились — за продолжительный период
простоя ни одного. Это сильно указывает, что заметная часть наблюдавшихся в
сессии переподключений была **спровоцирована самими действиями диагностики**
(рестарты `wireplumber`, `connect`), а органическая частота разрывов — **низкая**.

**Настоящую причину** (host-side или device-side) даст только код `Reason:` из
`btmon`, пойманный при **пассивном** наблюдении. Так как разрыв редкий, для этого
поставлен/рекомендован **фоновый захват** (см.
[«Фоновый захват»](#фоновый-захват-поставил-и-забыл-когда-разрыв-редкий)):
он сам запишет причину, когда разрыв случится, без ручного караула. Это следующий
шаг — ждём `Reason:`.

## Сделанные изменения (подробный журнал)

Здесь зафиксировано **всё**, что было изменено или запущено в ходе диагностики,
чтобы любую машину можно было вернуть в исходное состояние и чтобы было понятно,
какие наблюдения могли быть искажены этими действиями.

### Машина A — `thinkpad` (Intel AX211 / Ulanzi)

**Изменений не вносилось.** Выполнялись только read-only команды (чтение
`/sys/.../power/*`, `bluetoothctl info`, `journalctl`, `lsusb`, `hciconfig`,
delta-тест — он тоже только читает `runtime_suspended_time`).

Фикс `btusb enable_autosuspend=0` был **предложен, но НЕ применён**, потому что
delta-тест опроверг теорию автосуспенда. Файла `/etc/modprobe.d/btusb-no-autosuspend.conf`
на этой машине **нет**.

Контроллер `power/control` — не трогали (остался `auto`).

### Машина B — `projector` (Realtek 0bda:b850 / HOS301)

#### Изменение 1 — ПОСТОЯННОЕ: файл конфигурации WirePlumber

**Что:** создан новый файл
`~/.config/wireplumber/bluetooth.lua.d/51-no-suspend.lua`
(пользователь `spectator`).

**Назначение:** отключить усыпление BT-аудио-узлов
(`session.suspend-timeout-seconds = 0`).

**История файла (важно — было две версии):**

- **Версия 1 (неверная, неэффективная).** Правило матчило карту устройства
  (`device.name` = `bluez_card.*`). Настройка **не дошла до узла**
  (`pw-dump` показывал `suspend-timeout: <не задан>`), усыпление продолжалось.
- **Версия 2 (текущая на диске).** Правило исправлено на матч по **узлам**
  (`node.name` = `bluez_output.*` и `bluez_input.*`). Настройка села корректно
  (`pw-dump` показал `suspend-timeout: 0`). Именно этот вариант лежит в файле
  сейчас:

  ```lua
  table.insert(bluez_monitor.rules, {
    matches = {
      { { "node.name", "matches", "bluez_output.*" } },
      { { "node.name", "matches", "bluez_input.*" } },
    },
    apply_properties = { ["session.suspend-timeout-seconds"] = 0 },
  })
  ```

**Как создавался (для воспроизведения):**

```bash
mkdir -p ~/.config/wireplumber/bluetooth.lua.d
cat > ~/.config/wireplumber/bluetooth.lua.d/51-no-suspend.lua <<'LUA'
table.insert(bluez_monitor.rules, {
  matches = {
    { { "node.name", "matches", "bluez_output.*" } },
    { { "node.name", "matches", "bluez_input.*" } },
  },
  apply_properties = { ["session.suspend-timeout-seconds"] = 0 },
})
LUA
```

**Статус:** применено и активно, но **как причина разрывов НЕ подтверждено**
(с фиксом разрывы всё равно наблюдались). Само по себе изменение безвредно —
оно лишь оставляет BT-аудио-узел «бодрым».

**Как откатить:**

```bash
rm ~/.config/wireplumber/bluetooth.lua.d/51-no-suspend.lua
systemctl --user restart wireplumber   # ⚠️ рестарт сам вызовет разрыв/реконнект колонки
```

> Файл оставлен намеренно до конца диагностики: его удаление требует рестарта
> WirePlumber, а это ещё один искусственный разрыв, который испортит наблюдение.
> Убирать имеет смысл, только когда решено начисто вернуть исходное состояние.

#### Изменение 2 — РАЗОВОЕ: перезапуски WirePlumber

**Что:** `systemctl --user restart wireplumber` — выполнялось **несколько раз**
(при применении версии 1 и версии 2 файла выше).

**Следов в конфигах не оставляет**, НО каждый рестарт пересоздаёт аудио-узел
BT-устройства и **вызывает отвал + повторное поднятие** A2DP-транспорта, а иногда
и реконнект устройства.

**Последствие для диагностики:** часть переподключений, которые наблюдал
пользователь во время сессии, **вероятно, спровоцирована этими рестартами** —
по времени они совпали. Это искажение данных, а не органическая проблема.

#### Изменение 3 — РАЗОВОЕ: принудительные подключения

**Что:** `bluetoothctl connect 41:42:AB:F8:E2:1C` — выполнялось **несколько раз**.

**Следов не оставляет.** Если устройство было отключено — поднимало связь; тоже
могло добавить «событий подключения» в наблюдения.

#### Что на `projector` НЕ трогали

- Контроллер `power/control` — остался `auto` (исходное).
- `/etc/modprobe.d/` — не создавали (btusb-фикс не ставили).
- Настройки bluez / `/etc/bluetooth/` — не меняли.
- TLP / питание — не трогали.

## Приложение — быстрая шпаргалка команд

```bash
# --- ВВОДНЫЕ ---
bluetoothctl devices
bluetoothctl info AA:BB:CC:DD:EE:FF
hciconfig -a 2>/dev/null | grep -E "hci|BD Address|UP RUNNING"
lsusb | grep -i blue

# --- КОД ПРИЧИНЫ РАЗРЫВА ---
sudo btmon | grep --line-buffered -A3 -iE "disconnect complete|device disconnected"
#   если btmon падает "buffer overflow":
sudo btmon -w /tmp/bt.snoop           # Ctrl-C после разрыва
sudo btmon -r /tmp/bt.snoop 2>/dev/null | grep -A3 -iE "disconnect complete"
#   если разрыв редкий — фоновый захват "поставил и забыл":
sudo systemd-run --unit=btmon-cap --collect btmon -w /var/log/bt-cap.snoop   # старт
sudo btmon -r /var/log/bt-cap.snoop 2>/dev/null | grep -A3 -iE "disconnect complete"  # декод
sudo systemctl stop btmon-cap; sudo rm -f /var/log/bt-cap.snoop              # стоп+очистка

# --- АВТОСУСПЕНД КОНТРОЛЛЕРА (delta-тест) ---
BT=$(readlink -f /sys/class/bluetooth/hci0/device/..)
echo "control=$(cat $BT/power/control) status=$(cat $BT/power/runtime_status)"
x=$(cat $BT/power/runtime_suspended_time); sleep 60; y=$(cat $BT/power/runtime_suspended_time); echo "delta=$((y-x))ms"

# --- АУДИО-УЗЕЛ (колонки/наушники) ---
pw-dump 2>/dev/null | grep -i suspend-timeout
```
